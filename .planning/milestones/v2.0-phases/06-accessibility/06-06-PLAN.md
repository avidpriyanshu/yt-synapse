# Phase 6 Plan 6: Accessibility

**Objective:** Make yt-vault accessible and user-friendly for non-technical users by creating a settings UI, rewriting logging in plain language, building a first-run wizard, and auditing error messages.

**Success Criteria:**
1. Settings page accessible from Service Controls with configurable options
2. All log messages written in plain English (no jargon, complete sentences)
3. First-run wizard guides users through initial setup
4. Error messages explain what went wrong and how to fix (no stack traces)

## Task 1: Create Settings UI Page

**Type:** auto

**Description:** Build a markdown-based Settings page in Obsidian that allows users to configure scraper options without editing config files.

**Implementation:**
- Create `obsidian-vault/Service Controls/Settings.md`
- Sections for:
  - Scraper Settings (enable/disable, auto-start toggle)
  - Logging Preferences (verbose vs. summary)
  - Channel Tagging Preferences
  - Data Management (view logs, reset data)
- Use Obsidian callouts for each section
- Include clear instructions for each setting
- Settings backed by `.planning/settings-user.json`

**Done Criteria:**
- Settings page exists and is accessible from Service Controls
- Each setting has clear description for non-technical user
- Settings page is readable in Obsidian and doesn't require code editing
- Settings structure validated to match config loader expectations

## Task 2: Rewrite Logging in Plain Language

**Type:** auto

**Description:** Audit and rewrite all logging messages to be clear, jargon-free, and complete English sentences.

**Implementation:**
- Update logger.js to format messages as:
  - Complete sentences with context
  - Human-readable action descriptions
  - Clear outcomes (success/failure)
  - Avoid: abbreviations, error codes, JSON, timestamps in milliseconds
- Example: `Successfully added video "Jet Engine 101" from TechExplained`
- Apply to all log levels (info, warn, error)
- Keep debug logs technical (for developers) if needed

**Done Criteria:**
- All user-facing log messages are plain English
- No abbreviations or jargon in standard logs
- Messages explain what happened and why
- Debug logs still available for technical troubleshooting
- Logs tested with sample scraping operations

---

## Task 3: Build First-Run Wizard

**Type:** auto

**Description:** Create a first-run wizard that guides new users through vault setup.

**Implementation:**
- Detect first run: Check if `obsidian-vault/videos/` directory exists
- Create `obsidian-vault/Service Controls/First Run Setup.md`
- Wizard steps:
  1. Welcome message explaining yt-vault purpose
  2. Channel interests selection (suggest example categories)
  3. Logging preference choice (verbose or summary)
  4. Auto-start preference (yes/no)
  5. Setup confirmation and next steps
- Save choices to `.planning/settings-user.json`
- Show wizard automatically on first vault open
- Markdown-based UI (no CLI)

**Done Criteria:**
- First-run detection works correctly
- Wizard page created and formatted for readability
- User choices saved to settings
- Wizard shown on first vault open (conditional in main.js)
- Non-technical user can complete without guidance

---

## Task 4: Audit and Fix Error Messages

**Type:** auto

**Description:** Audit all error handling in the codebase and rewrite user-facing error messages in plain language.

**Implementation:**
- Review error handlers in: extractor.js, resolver.js, harvester.js, processor.js, reviewer.js
- Rewrite error messages to format: `{What went wrong} — {Why it happened} — {How to fix}`
- Examples:
  - Bad: `ECONNREFUSED: connection refused`
  - Good: `Couldn't reach YouTube — check your internet connection. Try again in a moment.`
  - Bad: `TypeError: Cannot read property 'match' of undefined`
  - Good: `The video link doesn't appear to be valid. Check the URL and try again.`
- Keep technical stack traces for debug/logging only
- Provide recovery actions (retry, skip, report)

**Done Criteria:**
- All user-facing errors audited and documented
- Plain language error messages implemented
- Stack traces not shown to users
- Error messages include recovery guidance
- At least 10 error cases improved
- Tested with sample error scenarios
