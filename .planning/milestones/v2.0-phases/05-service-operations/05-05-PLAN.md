# Phase 5, Plan 05: Service & Operations Implementation

**Objective:** Create a centralized logger module, implement service controls UI, add logging throughout the pipeline, and enable auto-start capabilities for non-technical users.

**Vision:** Users can start/stop scraper and view logs without terminal knowledge.

---

## Context

### Reusable Patterns
- `main.js` already has `logPhase()` and `reportFile()` functions
- Child process spawning familiar in codebase
- Atomic file writes established

### Integration Points
- Service controls page in Obsidian vault
- Logger module hooks into all phases (watcher, extractor, resolver, harvester, processor, generator)
- Backend process management via child_process

---

## Tasks

### Task 1: Create Centralized Logger Module
**Type:** auto | **Done Criteria:** logger.js exports with INFO/WARN/ERROR levels, timestamped entries format, daily log file rotation

Create `/logic-engine/logger.js` module that:
- Exports `log(level, phase, action, detail)` function
- Supports levels: INFO, WARN, ERROR
- Format: `[2026-04-24T14:30:45Z] [EXTRACT] [INFO] Added video: title (id)`
- Appends to `.planning/logs/run-YYYY-MM-DD.log` (daily rotation)
- Non-blocking append writes

**Files affected:**
- Create: `logic-engine/logger.js`

---

### Task 2: Integrate Logger Throughout Pipeline
**Type:** auto | **Done Criteria:** All phases (extractor, resolver, harvester, processor, generator) call logger module; no direct console.log for operational events

Update files to use logger module:
- `logic-engine/engine.js` — log video extraction success/failures
- `logic-engine/reviewer.js` — log validation results
- `logic-engine/processor.js` — log topic extraction, channel creation
- `logic-engine/main.js` — log scrape start/completion

Replace operational `logPhase()` calls with logger module calls. Keep structural logging for debugging.

**Files affected:**
- Modify: `logic-engine/engine.js`, `logic-engine/reviewer.js`, `logic-engine/processor.js`, `logic-engine/main.js`

---

### Task 3: Create Service Controls UI Page
**Type:** auto | **Done Criteria:** Service controls page in obsidian-vault with Start/Stop/View Logs buttons, status indicator polling backend

Create `obsidian-vault/99-Service-Controls.md` with:
- "Start Scraping" button (triggers `node logic-engine/main.js` via fetch to backend)
- "Stop Scraping" button (kills running process)
- "View Logs" button (displays today's log file content)
- Status indicator: "Scraper: Running" / "Scraper: Stopped" (polls every 2s)
- Uses Obsidian callouts + embedded dataviewjs

Create backend endpoint at `logic-engine/server.js`:
- `POST /start` — spawn scraper child process, return process ID
- `POST /stop` — kill running process
- `GET /logs` — return today's log file content
- `GET /status` — check if process running

**Files affected:**
- Create: `obsidian-vault/99-Service-Controls.md`
- Create: `logic-engine/server.js`

---

### Task 4: Implement Auto-Start Feature
**Type:** auto | **Done Criteria:** Config field `autoStartScraper` readable; Obsidian plugin hook calls scraper on vault load (non-blocking)

Add to config:
- Create `logic-engine/config.json` with `autoStartScraper: false` (default off)
- Create `obsidian-vault/.obsidian/plugins/yt-vault-service/main.js` hook for vault-open event
- Hook spawns scraper if `autoStartScraper: true`, non-blocking (no await)

**Files affected:**
- Create: `logic-engine/config.json`
- Create: `obsidian-vault/.obsidian/plugins/yt-vault-service/main.js`

---

### Task 5: Add Logging to Error Cases
**Type:** auto | **Done Criteria:** All error paths log ERROR level with retry count, network status, exception message

Update error handling in:
- `logic-engine/engine.js` — log network timeouts, invalid metadata
- `logic-engine/reviewer.js` — log validation failures
- `logic-engine/processor.js` — log topic extraction errors

Format: `[PHASE] [ERROR] Reason: {message} (retry {N}/{MAX})`

**Files affected:**
- Modify: `logic-engine/engine.js`, `logic-engine/reviewer.js`, `logic-engine/processor.js`
